# Phase 7: CI 連携 + エージェントワークフロー

## 1. 目的

運用者向けに GitHub Actions での差分ベース選択撮影のレシピを、AI エージェント向けに撮影結果を使った視覚検証手順を、それぞれ独立したドキュメントとして整備する。対応する仕様は overview.md §6（GitHub Actions 連携）・§7（AI エージェント連携）。

## 2. スコープ

### 含むもの

- `docs/github-actions.md`: PR diff から対象 system spec ファイルを抽出し、`SCREENSHOT_TESTS_FILE` 経由で gem に渡し、成果物を artifact 化するまでのレシピ。
- `docs/agent-workflow.md`: 撮影済み PNG 集合をエージェントが読んで視覚検証するための手順。

### 含まないもの（Non-goals）

- diff → 対象テストファイル抽出ロジック自体の実装（シェルスクリプトや Actions ワークフロー本体のコードを gem に持ち込むことは overview.md §1.3 で明示的に非対象。本フェーズはあくまで「レシピ」を文書として示すに留め、実行可能な CI 設定ファイルを本 gem リポジトリに追加するものではない）。
- 「ビューのみ変更で spec 不変」PR を拾うための変更ファイル→対象テストマッピング（TIA・規約マッピング）の設計（overview.md §12 の将来課題）。
- README（P6 の責務）との重複記述。有効化モデルや gist 移行の話はここでは扱わない。

## 3. 前提・依存

- **先行フェーズ**: P4（TargetListPolicy + 対象リスト受け取り）に依存する。対象リスト受け渡し口 `SCREENSHOT_TESTS_FILE`（および補助的な `SCREENSHOT_TESTS`）が確定していることが前提であり、これに基づいて CI レシピの受け渡し部分を記述する。
- **並行可能フェーズ**: P6（README + 移行手順）と並行して進めてよい。README とは参照関係のみで実装的な依存はない。
- **外部前提**: GitHub Actions 上で PR の base ブランチとの diff が取得できること。エージェント側の前提は Claude Code 等、ファイルシステム上の PNG を読める AI エージェント環境であること。

## 4. 実装方針（＝執筆内容）

### docs/github-actions.md

- レシピの骨子（overview.md §6 の骨子どおり）:
  1. PR の base ブランチとの diff を取得し、`spec/system/*_spec.rb` にマッチする変更ファイルのみ抽出する。
  2. 抽出結果を `tmp/screenshot_targets.txt` に改行区切りで書き出す（該当なしの場合は空ファイルで可）。
  3. system spec 実行時に `SCREENSHOTS=1` と `SCREENSHOT_TESTS_FILE=tmp/screenshot_targets.txt` を環境変数として渡す。
  4. 生成された `tmp/screenshots/**` を CI の artifact としてアップロードし、人間・エージェントが後から参照できるようにする。
- 上記をワークフロー YAML の意図が伝わる記述で示す（実行可能な `.github/workflows/*.yml` そのものの提供が目的ではなく、利用者が自分のリポジトリの CI に組み込める程度の具体性で書く）。
- 明記すべき注意点:
  - 「ビューだけ変更で system spec ファイル自体は不変」の PR ではこの方式は変更を拾えない（diff にテストファイルが出現しないため）。これは初版の既知の割り切りであり、対象はテストファイルパス指定のみであることの帰結であると明記する。
  - 対象なし（空ファイル）の場合、撮影 0 件になるのが正しい挙動であることを明記する（絞り込みが効いて何も撮らない、というのは異常ではない）。

### docs/agent-workflow.md

- gist の `screenshot-test` スキル相当の手順を、diff ベースの対象決定に書き直した版として記載する。
  1. テスト実行（`SCREENSHOTS=1` + 対象リストを指定した system spec 実行）。
  2. 出力された PNG をテスト単位・連番順に列挙する。
  3. 1 枚ずつ視覚的に検証する（レイアウト崩れ・意図しない表示・エラー画面の有無など）。
  4. テスト単位でレポートする（どのテストのどの操作直後に何が確認できたか）。
  5. 後片付け（一時生成物の扱い、artifact との対応関係）。
- 出力レイアウト（overview.md §3.8: `tmp/screenshots/{GroupName}/{example_name}/{NNN_action_detail}.png`）と手順の記述を整合させる。特に「ファイル名がアクションを説明する」ため、エージェントが連番とファイル名から操作順序を追える、という読み方の指針を明記する。
- 「対象を絞ることでエージェントが読む枚数を現実的な規模に抑える」という設計意図（overview.md §7）を、手順の前提として簡潔に述べる。

## 6. 受け入れ条件（レビュー観点）

- [ ] `docs/github-actions.md` のレシピが overview.md §6 の骨子（diff 抽出 → ファイル書き出し → ENV 経由で渡す → artifact 化）と一致している。
- [ ] 「ビューのみ変更で spec 不変」の割り切りと、「対象なし = 撮影 0 件が正しい」旨が明記されている。
- [ ] `docs/agent-workflow.md` の手順が overview.md §3.8 の出力レイアウトと整合している（ディレクトリ構成・ファイル名規則の説明が実際の仕様と一致）。
- [ ] `docs/agent-workflow.md` の手順が overview.md §7 の設計意図（対象を絞ることでエージェント可読な規模に抑える）を反映している。
- [ ] minitest 前提の記述が一切含まれていない。

## 7. テスト観点（レビュー観点/追試手順）

- **レビュー観点**:
  - `docs/github-actions.md` のレシピどおりに CI ワークフローを組んだ場合、対象リストのファイルと `tmp/screenshots/**` の対応関係が説明どおりになるか（overview.md §3.5 のパス正規化前提と矛盾しないか）を確認する。
  - `docs/agent-workflow.md` の手順が、実際の出力ディレクトリ構造（テストごと・連番・アクション名入りファイル名）をそのまま利用する形になっており、追加の変換や前処理を要求していないかを確認する。
- **追試手順**:
  - 小さな PR（1 spec ファイルのみ変更）を想定したケースで、diff 抽出からレシピを辿り、`tmp/screenshot_targets.txt` に想定どおりのパスが 1 行だけ書き出される、という手順がドキュメント上で追えるかを検証する。
  - 変更が spec ファイルに一切かからない PR（ビューのみ変更）を想定したケースで、「対象なし・撮影 0 件」という割り切りの記述が読み手に誤解なく伝わるかを確認する。
  - エージェントワークフローについては、少数（3〜5 枚程度）の PNG 集合を想定したウォークスルーで、手順の各ステップ（列挙 → 視覚検証 → レポート → 後片付け）が過不足なく踏めるかをレビューする。

## 8. リスク・決定事項

- 本フェーズ固有の新規決定事項はない（overview.md §11 の決定事項はいずれも P1〜P4 で確定済みであり、本フェーズはそれらを前提に文書化するのみ）。
- リスクとして、P4 の対象リスト正規化ルールの細部が変わった場合、`docs/github-actions.md` の diff 抽出・書き出し例の記述が古くなる可能性がある。overview.md の該当節を参照する形で記述し、独自の正規化ルール解説を書き足さないことでズレのリスクを抑える。
